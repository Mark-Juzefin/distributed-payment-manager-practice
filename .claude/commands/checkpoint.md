# Checkpoint Session Context

Захопити нетривіальний стан із поточної розмови у файли активної фічі, щоб
наступний агент міг продовжити без втрати контексту.

## Task

1. **Знайти активну фічу**
   - Прочитати `CLAUDE.md` → знайти "Active feature" посилання на
     `docs/features/...`
   - Прочитати `README.md` цієї фічі
   - Перелічити `plan-subtask-N.md` файли у тій же папці

2. **Вирішити що захопити** з поточної сесії, у порядку пріоритету:

   a. **Архітектурні рішення / constraints, що випливли** — закріпити у
      `## Notes` (короткі булети) і/або у memory (якщо кросс-фічеве)
   b. **Завершені підзадачі** — перемкнути чекбокс `- [ ] → - [x]`
   c. **Mid-flight progress** на підзадачі без plan-файла — створити або
      оновити `plan-subtask-N.md` з тим, що зроблено і що лишилось
   d. **Інші значущі події** (рефакторинг готов, задачу відкладено,
      знайдено blocker) — додати запис у `## Session Log` з сьогоднішньою
      датою

   **Пропустити те, що**:
   - Уже видно з коду (`git log`, `git status`, читання файлів)
   - Тривіальна conversational переписка
   - One-off питання, які не вплинуть на майбутню роботу

3. **Застосувати зміни напряму, без підтвердження** — не питати користувача,
   не показувати numbered list для approval. Користувач уже викликав
   `/checkpoint` — це і є дозвіл.

   - Додати запис у `## Session Log` (створити секцію якщо немає). Формат:
     ```
     ### YYYY-MM-DD — короткий title
     - Bullet 1 (рішення/результат, не narration)
     - Bullet 2 (що відкладено або pending — з причиною)
     ```
   - Перемкнути чекбокси у `## Tasks`
   - Додати булети у `## Notes` для закріпленої інфи
   - Оновити/створити `plan-subtask-N.md` якщо є mid-flight контекст
   - Зберегти memory якщо є кросс-фічеве рішення

4. **Після запису** сказати користувачу коротко з чого починати наступну сесію:
   - Поточна підзадача + plan-файл (якщо є)
   - Відкриті питання / pending decisions
   - Будь-яка відкладена робота, що потребує окремого планування

## What NOT to capture

- File paths, line numbers, function names, що уже у коді
- Детальний step-by-step що саме редагувалось (git diff це показує)
- Verbose narration debug-процесу — важливий лише висновок
- Будь-що, що покриває `git log` / `git blame`

Барьер: "чи зможе свіжий агент, читаючи це, продовжити роботу — чи це
просто redundant лог минулого?". Зберігати тільки перше.

## Usage

```
/checkpoint
```

Викликати в кінці сесії або перед перемиканням на іншого агента.
