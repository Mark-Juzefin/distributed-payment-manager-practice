


Спробував pglogrepl — наочно побачив, як створюється replication slot, як із pgoutput парситься WAL, і як воркер веде позицію (LSN) та шле heartbeat/ack. Але для “просто прочитати INSERT-и з outbox” тут дуже багато низькорівневої, ручної роботи.
Натрапив на проблему зі створенням/використанням ReplicationSlot: довелося додати костиль — на старті дропати слот і створювати заново, щоб уникати залипання на неконсистентному стані (наприклад, коли publication/таблиця ще не існували під час першого старту).
Окремий біль — ручне управління LSN: треба правильно обирати, що саме ack-ати (і коли), інакше або ризик втрати подій, або ризик роздування WAL через slot retention.
Висновок: pglogrepl — класний “підкапотний” інструмент, який ідеально підходить як база для бібліотек/фреймворків/інфра-компонентів, але для прикладного продакшн-коду зазвичай хочеться рівень абстракції вище (Debezium / готовий CDC-компонент / polling-outbox).

Щоб спростити задачу, але все ще залишитися на рівні абстракції CDC переключуся на pgcapture - який усі ці незручні речі робить під капотом: replication connection, тримає slot/LSN, парсить повідомлення logical replication
і віддає уже нормальні нормальні CDC-івенти.
Правда це уже ен просто бібліотека, потрібно ще запустити їх server side components, гайд:
https://github.com/replicase/docker-pgcapture
Але схоже цей інструмен не підійде, бо pgcapture вимагає серйозну інфраструктуру

Повний стек компонентів:
1. Apache Pulsar — окремий message broker (окрім Kafka, який вже є)
2. pg2pulsar — мост PG WAL → Pulsar
3. controller — gRPC контролер реплікації
4. gateway — gRPC gateway для consumer API
5. agent — для dump операцій
6. pulsar2pg — sink реплікація
7. Custom PostgreSQL image з їхнім конфігом



Ще трішки пошукав готові інструменти і зрозумів, що реалізувати це власноруч не так і складно
